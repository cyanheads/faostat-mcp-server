# 0.1.3 — 2026-06-19

## Fixed

- **`faostat_query_observations` mid-size dead band** — results of 51–~600 rows no longer silently drop rows. The spill *decision* (`total > INLINE_PREVIEW_ROWS`) used a row count while the spill helper materializes a canvas table only once serialized output crosses a *character* budget (`PREVIEW_CHARS = 100_000`, ~600 rows). In the gap, `shouldSpill` was true but `spillover()` returned `{ spilled: false }`, and the handler re-capped at `limit` (default 200) and emitted an "Enable DataCanvas" notice even when DataCanvas was already on. The handler now returns the not-spilled `previewRows` — which the helper drains to the **complete** set under the char budget — inline, so no row is dropped and no `canvas_id` is required. The `limit`-capped inline page is reached only for genuinely small results (≤ 50 rows) or when a canvas op fails outright. [#1](https://github.com/cyanheads/faostat-mcp-server/issues/1)
- **`faostat_commodity_profile` same dead-band notice** — when canvas is on but the merged set fits under the char budget, the notice and `format()` no longer advise enabling an already-on canvas; the rankings and trend count already cover the full result. [#1](https://github.com/cyanheads/faostat-mcp-server/issues/1)
- **Dead and mismatched error-contract reasons** — removed unreachable `errors[]` entries and aligned the rest with what the handlers and SQL gate actually emit, so each tool's advertised failure modes match reality. [#2](https://github.com/cyanheads/faostat-mcp-server/issues/2)

## Changed

- **`faostat_query_observations` `errors[]`** — dropped `unknown_domain` (the handler throws `domain_not_indexed` for any non-selected domain) and `empty_result` (zero matches return an empty result + notice, never throw). Kept `domain_not_indexed`, `index_not_ready`, `canvas_disabled`. [#2](https://github.com/cyanheads/faostat-mcp-server/issues/2)
- **`faostat_resolve_codes` `errors[]`** — dropped `no_match` (a miss returns empty matches + notice). Kept `unknown_domain`, `index_not_ready`. [#2](https://github.com/cyanheads/faostat-mcp-server/issues/2)
- **`faostat_dataframe_query` gate-reason remap** — `queryStaged` now normalizes every SQL-validity gate `McpError` (`non_select_statement`, `multi_statement`, `plan_operator_not_allowed`, `denied_function`, `denied_function_in_plan`, and the identifier reasons, sourced from the exported `SQL_GATE_REASONS`) to the stable, declared `invalid_sql`, preserving the gate's message. `missing_table` and `system_catalog_access` keep their own declared reasons. The tool's declared set is now exactly `canvas_disabled`, `missing_table`, `system_catalog_access`, `invalid_sql` — all reachable, none leaking an undeclared `data.reason`. [#2](https://github.com/cyanheads/faostat-mcp-server/issues/2)
