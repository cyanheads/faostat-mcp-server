# 0.1.7 — 2026-07-13

## Added

- **`truncated` + `staged_row_count` — `faostat_query_observations` and `faostat_commodity_profile` output fields.** `truncated` is `true` when the staged set hit the 50,000-row (`STAGE_MAX_ROWS`) staging cap; `staged_row_count` reports the rows actually staged. Both surfaced in `structuredContent`, `format()`, and the enrich notice. ([#9](https://github.com/cyanheads/faostat-mcp-server/issues/9))
- **`invalid_year_range` — typed `InvalidParams` error on `faostat_query_observations`.** A reversed `year_start > year_end` now fails with actionable recovery instead of a silent generic no-match. ([#12](https://github.com/cyanheads/faostat-mcp-server/issues/12))

## Fixed

- **50,000-row staging cap presented as the complete result.** `faostat_query_observations` and `faostat_commodity_profile` staged at most `STAGE_MAX_ROWS` rows but disclosed no cap, so a capped canvas table — and `commodity_profile`'s producer ranking and `trend_points` — read as complete. Both now flag `truncated`, mark the staged table/profile a partial prefix, and direct the caller to partition by year (`year_start`/`year_end`) for the full series. ([#9](https://github.com/cyanheads/faostat-mcp-server/issues/9))
- **Empty code arrays broadened the query to the whole dimension.** A defined-but-empty `area_codes` / `item_codes` / `element_codes` was dropped by the mirror's `.length` guard and silently queried every code in that dimension; it now short-circuits to a zero-match with a distinct notice before any mirror call. ([#12](https://github.com/cyanheads/faostat-mcp-server/issues/12))
