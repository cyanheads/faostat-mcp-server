# 0.1.10 ‚Äî 2026-07-13

## Changed

- **Data-quality flag legend widened.** The legend in the server `instructions`, `faostat_query_observations`, `faostat_dataframe_query`, and `faostat_commodity_profile` grows from a closed `A/E/I` list to the fuller FAOSTAT set ‚Äî `A`=Official, `B`=time-series break, `E`=Estimated, `I`=Imputed, `M`=Missing (value cannot exist), `T`=Unofficial, `X`=from an international organization, plus others FAOSTAT defines per domain ‚Äî and is reframed as an open set: treat any unrecognized flag as informational, never assume it's official. `faostat_resolve_codes` carries no flag legend and is unaffected. ([#6](https://github.com/cyanheads/faostat-mcp-server/issues/6))

## Fixed

- **Aggregate classification for sub-threshold roll-up areas.** `faostat_resolve_codes` and `faostat_query_observations` disagreed on whether a code below `AGGREGATE_AREA_CODE_THRESHOLD` (5000) was a country or an aggregate; China (`351`, a roll-up of mainland + Taiwan + Hong Kong + Macao) and the TCL trade roll-up "China (excluding intra-trade)" (`265`) passed as countries and could be double-counted alongside their member countries in a naive `SUM`. A shared `isAggregateAreaCode()` predicate plus a curated `AGGREGATE_AREA_CODES` deny-set (`{265, 351}`) in `types.ts` is now the single source of truth ‚Äî `classifyArea()` and the default aggregate-exclusion `WHERE` clause in `faostat-mirror.ts` both call it, and `DimensionsStore.toResolved()` derives `kind` at read time from the code rather than a stored column, so already-ingested mirrors reclassify with no re-sync. ([#4](https://github.com/cyanheads/faostat-mcp-server/issues/4))
