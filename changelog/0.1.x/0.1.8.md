# 0.1.8 — 2026-07-13

## Changed

- **Spilled and preview rows now order by `year`.** When `faostat_query_observations` or `faostat_commodity_profile` spills a large result to a canvas table, the stream sorts `ORDER BY year` (was `area, item, element, year`) — so the staged rows, their preview, and which rows fall inside a truncated 50,000-row prefix are year-ordered. The output declares no ordering and `faostat_dataframe_query` re-sorts, so downstream analysis is unaffected. ([#3](https://github.com/cyanheads/faostat-mcp-server/issues/3))
- **`totalCount` past the staging cap is now a floor, not the exact match count.** With the `COUNT(*)` gone, a result that stages fully still reports an exact `totalCount` (derived from the drained stream); one that exceeds the 50,000-row (`STAGE_MAX_ROWS`) cap reports that cap as a floor alongside `truncated: true`, rather than the exact upstream total. ([#3](https://github.com/cyanheads/faostat-mcp-server/issues/3))

## Fixed

- **Event-loop-blocking query path.** Every `faostat_query_observations` / `faostat_commodity_profile` call ran an unbounded `COUNT(*)` over the whole matched set to size the inline-vs-spill decision, and each spill sorted the stream by four unindexed `TEXT` columns (`area, item, element, year`) — synchronous scans on better-sqlite3's main thread that stalled the event loop on large domains. The `COUNT(*)` is now a `LIMIT+1` overflow probe (a `totalIsExact` flag reading just past the inline budget), the spill sort is `ORDER BY year`, and two composite indexes — `(element_code, year)` and `(item_code, element_code, year)`, created via `CREATE INDEX IF NOT EXISTS` on open() — back the seek and sort and apply to already-synced mirrors on the next startup with no re-sync. ([#3](https://github.com/cyanheads/faostat-mcp-server/issues/3))
